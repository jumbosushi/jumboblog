While researching the differences between Cypress and Playwright, I was surprised by the lack of articles covering their architectures. Most of the articles were either not in-depth enough or slightly outdated. This is my attempt to document what I learned. We'll cover Cypress first, then Playwright, then compare both in terms of performance and flakiness. 

# Cypress architecture 

![cypress architecture](/images/2026-04-10-cypress/cypress-architecture.png)

Cypress has three major components (based on `v15.13.0`):
- Cypress Server: Orchestrates the whole Cypress workflow (e.g. file management, browser lifecycle, [task execution](https://docs.cypress.io/api/commands/task), etc.)
- Browser: Electron or Chrome where it runs three windows (Cypress App, Test driver, and the App Under Test (AUT))
- Proxy: Intercepts all HTTP requests from the browser

Cypress server and proxy are part of the Electron app (which was surprising to me, but here's the [GitHub issue](https://github.com/cypress-io/cypress/issues/3899) about it). On browser start, the server sets the proxy via the browser configuration. The [ `--proxy-server`](https://www.chromium.org/developers/design-documents/network-settings/) flag is used in Chrome and [`session.SetProxy`](https://www.electronjs.org/docs/latest/api/session#sessetproxyconfig) is used for Electron. 

The server uses SQLite to store state. This includes both `cy.session()` and creating a separate `.db` file per test suite.

## Test driver & AUT

After Cypress app initializes in the browser, it loads two iframes:
1. Test driver: Executes the `cy` test code
2. App under test (AUT): User application being tested.

Cypress internally uses [Mocha](https://docs.cypress.io/app/references/bundled-libraries#Mocha) as the test runner (where `.describe()` and `.it()` API are from). For element selectors, it uses [jQuery](https://docs.cypress.io/api/utilities/$). Although the Test driver iframe runs the test code, it isn't executed synchronously. The test driver implements its own runtime where tasks are enqueued and executed asynchronously (this is well documented in [the doc](https://docs.cypress.io/app/core-concepts/introduction-to-cypress#Commands-Are-Asynchronous)). 

Why does it use iframe in the first place? From what I can find online, it's a product of history. [Selenium 1](https://www.selenium.dev/documentation/legacy/selenium_1/) originally used a similar iframe injection architecture, but migrated to using [WebDriver Classic](https://developer.chrome.com/blog/test-automation-evolution) (predecessor of CDP - Chrome DevTools Protocol) in [Selenium 2](https://www.selenium.dev/documentation/legacy/selenium_2/upgrading/). WebDriver Classic serialized every DOM interaction into an HTTP request/response cycle which was a performance bottleneck. Cypress extended the original `<iframe>` architecture with a goal to reduce check-and-act cycle latency.

Normally, iframes with different origins can't interact with each other due to the same-origin policy. Cypress works around this by overwriting the origin of all three windows to be the same: app, test driver, and AUT.

## Proxy & Protocols

All HTTP requests from the browser are intercepted by the proxy. The proxy distinguishes between the Cypress App requests and AUT requests by an `X-Cypress-Is-AUT-Frame` HTTP header. This enables commands like `cy.intercept`:
```js
// spying and response stubbing
cy.intercept('POST', '/users*', {
  statusCode: 201,
  body: {
    name: 'Satoru Gojo',
  },
})
```

The way Cypress uses WebSocket and CDP is interesting and under-documented. WebSocket is used for tasks that specifically require the Server like [`cy.task()`](https://docs.cypress.io/api/commands/task) and [`cy.exec()`](https://docs.cypress.io/api/commands/exec). CDP, on the other hand, is used for the proxy. Cypress listens for the `Fetch.requestPaused` event then adds the AUT HTTP header appropriately. This is done in the CDP layer since the request source is distinguished by the `frameId` ([code](https://github.com/cypress-io/cypress/blob/80e20d26d9afb66d9ac336bac61e974ed9d181f8/packages/server/lib/browsers/cdp_automation.ts#L525-L537)):

```js
client.on('Fetch.requestPaused', async (params) => {
    if (await this._isAUTFrame(params.frameId)) {
      return this._continueRequest(client, params, {
        name: 'X-Cypress-Is-AUT-Frame',
        value: 'true',
      })
    }
    return this._continueRequest(client, params)
  })
```

Then there are operations where it uses both WebSocket and CDP. For example, when there's a call like `cy.screenshot()`:
1. Test Driver dequeues the `cy.screenshot()` step
2. Test Driver sends a request via websocket to the server 
3. Server sends a [`Page.captureScreenshot`]([Page.captureScreenshot](https://chromedevtools.github.io/devtools-protocol/tot/Page/#method-captureScreenshot)) CDP request to the browser

# Playwright Architecture

![playwright architecture](/images/2026-04-10-cypress/playwright-architecture.png)

Playwright's architecture is simpler compared to Cypress (based on `v1.59.1`). When you're running tests in a single machine, these are the roughly three components:
- Dispatcher: The main process that orchestrates the whole workflow
- Worker: Forked from Dispatcher, and executes the tests 
- BrowserContext: Manages independent browser sessions, each with its own state (e.g. localStorage)

Dispatcher and Worker communication is done with the JSON-RPC over the [Node.js process events](https://nodejs.org/api/process.html#process-events). Many articles mention that Playwright uses WebSocket for worker orchestration, but this only applies to the [client-server](https://playwright.dev/docs/docker#remote-connection) use case. Workers control browser sessions using CDP for Chrome, and Playwright's own protocol implementations for Firefox and WebKit.

Worker both executes the test code and maintains the CDP connections. It serializes the test code to CDP as it executes sequentially via `async`/ `await`.

# Architecture comparisons

## Performance 

Many articles claim that Playwright is more performant than Cypress. Cypress has more overhead: Electron, a proxy, and iframes. It also bundles Mocha for tests, jQuery for DOM queries, Vue.js for the in-browser Cypress App, [Bluebird](https://github.com/cypress-io/cypress/issues/33032) for Promises, [Webpack](https://github.com/cypress-io/cypress/issues/31751), and probably more. The Cypress App is [rendered](https://docs.cypress.io/cloud/features/test-replay#Runner-UI) in headless mode unless you disable it with `--no-runner-ui` or use their Cloud service.

Playwright executes much less JavaScript on top of relying entirely on CDP for browser interactions. The biggest performance gain would come from sharding & single-machine parallelization (using [BrowserContext](https://playwright.dev/docs/api/class-browsercontext) and [storageState](https://playwright.dev/docs/auth#moderate-one-account-per-parallel-worker)).

## Flakiness

Many articles claimed that Playwright is less flaky, but this is more of a nuanced discussion. 
Cypress has the [retry-ability](https://docs.cypress.io/app/core-concepts/retry-ability) feature, but [so does Playwright](https://playwright.dev/docs/test-retries). Similarly, Playwright's [auto-wait](https://playwright.dev/docs/actionability) gets lots of praise, but Cypress also has a similar [`ensure` feature](https://github.com/cypress-io/cypress/blob/89a16f3538abccb0dc12ecdeb0b54193238462c4/packages/driver/src/cy/actionability.ts#L424-L431) in the code. In general, the real risk of flakiness depends mostly on how the test code is written. 

With that being said, debugging flaky tests seems substantially easier with Playwright. It can produce a self-contained `.zip` [trace file](https://playwright.dev/docs/trace-viewer) capturing network, console, and DOM snapshots that AI can consume easily. This would be helpful for investigating flaky CI test cases, particularly with tools like its [MCP server](https://github.com/microsoft/playwright-mcp) and [CLI](https://github.com/microsoft/playwright-cli) being available.

# Conclusion

For E2E testing, Playwright seems like a better fit for my use cases. Cypress's iframe and command-queue model predates `async`/`await`, and it prioritizes interactive debugging features over headless CI.  Let me know if you found this useful or noticed something wrong!
